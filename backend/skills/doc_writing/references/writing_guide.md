## 1. 문서 유형별 사용 기준

| 문서 유형 | 사용 시점 | 주요 독자 | 필수 포함 항목 |
|---|---|---|---|
| **PRD** | 신규 기능/제품 기획 시 | 개발, 디자인, QA | 배경, 요구사항, 일정 |
| **기술 기획서** | 시스템 설계 및 아키텍처 결정 시 | 개발팀 | 기술 스택, 구조도, API |
| **정책 문서** | 서비스 운영 규칙 수립 시 | 전 팀원 | 규칙, 예외 사항, 적용 일자 |
| **회의록** | 주요 결정 사항 기록 시 | 참석자 + 관련자 | 결정 사항, 액션 아이템 |
| **가이드라인** | 프로세스/방법론 정립 시 | 전 팀원 | 절차, 예시, FAQ |

## 2. 공통 문서 구조 원칙

### 2.1 문서 헤더 (필수)
모든 문서는 상단에 아래 메타 정보를 포함해야 합니다.

```markdown
# 문서 제목

> 작성일: YYYY-MM-DD  
> 버전: v1.0  
> 작성자: 이름  
> 상태: 초안 | 검토 중 | 확정 | 폐기
```

### 2.2 버전 관리
- `v0.x`: 초안 (공식 배포 전)
- `v1.x`: 첫 번째 공식 버전
- 마이너 수정: v1.0 → v1.1 (내용 추가/수정)
- 메이저 변경: v1.x → v2.0 (구조 전면 개편)

### 2.3 상태 표시
- **초안**: 작성 중, 내부 검토 전
- **검토 중**: 관련자 리뷰 진행 중
- **확정**: 승인 완료, 공식 적용
- **폐기**: 더 이상 유효하지 않음

## 3. 마크다운 작성 규칙

### 3.1 헤딩 구조
```
# 문서 제목 (H1 - 문서당 1개)
## 섹션 (H2)
### 하위 섹션 (H3)
#### 세부 항목 (H4 - 최대 깊이)
```

### 3.2 강조 사용 기준
- **굵게**: 핵심 용어, 중요 결정 사항
- *기울임*: 보조 설명, 예시
- `코드 형식`: 기술 용어, API 경로, 코드

### 3.3 목록 사용 기준
- **순서 없는 목록** (`-`): 순서가 없는 항목 나열
- **순서 있는 목록** (`1.`): 단계별 절차, 우선순위가 있는 항목
- **체크리스트** (`- [ ]`): 할 일 목록, 수용 기준

### 3.4 표 사용 기준
| 적합한 경우 | 부적합한 경우 |
|---|---|
| 여러 항목 비교 | 2개 항목만 비교할 때 |
| 구조화된 데이터 | 단순 나열 |
| 3개 이상 속성 정리 | 1-2개 속성 |

## 4. 섹션별 작성 지침

### 4.1 배경 및 목적 작성법
- **나쁜 예**: "이 기능은 유저 편의를 위한 것입니다."
- **좋은 예**: "현재 결제 완료 후 영수증 확인을 위해 평균 3단계의 메뉴 탐색이 필요합니다. 이로 인해 CS 문의의 23%가 영수증 관련입니다. 영수증 바로 확인 기능을 추가하여 CS 문의를 15% 감소시키는 것이 목표입니다."

### 4.2 요구사항 작성법
요구사항은 **측정 가능하고 검증 가능하게** 작성합니다.

- **나쁜 예**: "빠르게 로딩되어야 한다."
- **좋은 예**: "첫 화면 로딩 시간이 LCP 기준 2.5초 이내여야 한다. (측정: Lighthouse)"

### 4.3 사용자 스토리 형식
```
[사용자 유형]로서,
[목적/이유]를 위해,
[기능/행동]을 하고 싶다.
```

예시: "결제 완료 사용자로서, 구매 내역을 바로 확인하기 위해, 결제 완료 페이지에서 영수증을 즉시 볼 수 있고 싶다."

## 5. 리뷰 및 승인 프로세스

```
작성 → 자체 검토 → 관련자 공유 → 피드백 반영 → 최종 승인 → 배포
  ↑_____________________________|
         (피드백 루프)
```

### 5.1 자체 검토 체크포인트
- [ ] 문서 목적이 첫 단락에서 명확한가?
- [ ] 독자가 누구인지 고려했는가?
- [ ] 모든 약어와 전문 용어가 설명되어 있는가?
- [ ] 데이터/수치에 출처가 있는가?
- [ ] 액션 아이템에 담당자와 기한이 있는가?

## 6. 자주 하는 실수

| 실수 | 올바른 방법 |
|---|---|
| 수동태 남용 ("~되어야 한다") | 능동태 사용 ("~해야 한다") |
| 모호한 주어 ("우리는", "팀은") | 구체적 주어 ("개발팀은", "PM은") |
| 날짜 없는 결정 사항 | 모든 결정에 날짜 기록 |
| 구두 결정만 존재 | 회의 후 반드시 문서화 |
| "추후 논의" 방치 | 담당자와 기한 명시 |
